# 15-visual-reference-library

读取时机：当任务涉及视觉参考图、风格调研、Design DNA 提取、Style Frame 方向、审美策略沉淀、或用户提供外部设计参考链接时读取。若进入正式视觉生产，同时读取 `04-design-standards.md` 与 `11-aesthetic-anti-ai.md`。

## 目的

本文件是团队长期视觉参考库。它不用于简单收藏图片，也不是模板库、抄图库或风格套用清单，而用于把参考图、案例网页、设计风格理念和调研结论沉淀为可复用的设计判断。

后续所有与视觉调研、设计风格、审美策略、参考图分析相关的结论，都应沉淀到本文件或其拆分子文件中，并在对应项目版本的 evidence 中留下来源记录。

## 沉淀原则

- 参考图必须转译为设计经验，不能只保存链接。
- 参考图不能直接照搬照抄；任何借鉴都必须经过“经验抽象 → 当前项目改写 → Style Frame验证”。
- 每次沉淀至少回答：适用场景、可复用点、不可复用点、风险、生产规则。
- 每个风格必须先判断为通用型或专用型，再决定是否进入当前项目。
- 视觉语言要同时记录三层：Design System、Design Style、Visual Effects。
- 设计结论必须服务交付场景，不把流行风格直接套到培训、汇报、路演或研究报告中。
- 主品牌色、用户资料、真实证据和交付场景优先于参考图风格；流行趋势只能作为辅助判断。
- 正向经验和禁用模式同等重要；看到好的参考，也要记录它为什么不能机械照搬。
- 外部链接、截图、用户资料和临时素材必须标注来源、访问日期和权限状态。

## 禁止照搬协议

每次使用本库时，必须把参考图拆成三类信息：

| 类型 | 可以怎么用 | 禁止怎么用 |
| :--- | :--- | :--- |
| 结构经验 | 提取页面族谱、阅读路径、信息密度、组件关系 | 复制原图版式比例、模块位置和页面节奏 |
| 风格经验 | 提取字体气质、色彩策略、留白方式、视觉重量 | 直接替换成参考图配色、字体、图标、装饰和行业话术 |
| 风险经验 | 记录低对比、语义错配、专用场景、投屏问题 | 因为参考图“好看”而忽略当前受众、品牌和证据限制 |

使用参考图前必须先回答四个问题：

1. 当前用户资料中有什么真实品牌色、真实素材、真实数据或真实场景？
2. 参考图里真正可复用的是结构、节奏、组件语法，还是只是表层好看？
3. 哪些部分会削弱当前项目的可信度、可读性或场景适配？
4. 最终Style Frame如何证明它是“为当前项目设计”，而不是“像某张参考图”？

## 正向参考图深拆协议

只要某张参考图被判断为“正向、优秀、值得学习、可以吸收”，必须先完成《正向参考图深拆与经验总结表》。正向参考不是更容易使用的参考，而是更需要解释其设计成立原因的参考；否则团队只会复制表层热闹感、配色和装饰。

| 维度 | 必须回答 | 失败信号 |
| :--- | :--- | :--- |
| 配色逻辑 | 为什么这样配色；主色、辅助色、强调色、风险色、背景色分别承担什么语义；颜色比例如何服务情绪和可读性 | 只写“红蓝撞色/黄色醒目/高级灰”，没有语义和比例 |
| 文字协同 | 标题、正文、注释、标签、口播文案、引用和按钮如何分工；文字如何顺着画面走，而不是填满空位 | 只看文字内容，不分析文字在画面中的节奏和角色 |
| 字体系统 | 字体气质、字号层级、字重、字距、行距、倾斜/描边/阴影/高亮样式为什么适合当前受众和观看距离 | 只说“字体好看/大标题醒目”，没有层级和观看距离 |
| 版面规划 | 页面视觉重心、网格、留白、图文比例、边界、斜切/撕纸/卡片/箭头等组件如何组织阅读路径 | 把版面理解成对齐几个方块或复刻原图位置 |
| 页面切换 | 跨页主题、色带、标题容器、页眉页脚、图像槽、节奏变化如何延续；哪些页应该换气、哪些页应该保持连续 | 只拆单页，不说明多页之间如何过渡 |
| 设计假设 | 如果由当前团队来做，会提出哪2-3个方案；每个方案分别强化什么：冲击力、可信度、信息密度、互动感或品牌感 | 没有替代方案，默认照参考图走 |
| 动效方案 | 哪些内容应静止，哪些应分步出现，运动方向、时长、缓动、停顿和完成态如何服务讲述 | 只写“淡入/出现/切换”，没有讲述目的 |
| 交互与HTML | 如果产物是HTML，目录跳转、局部展开、滚轮/键盘/触摸、回退/重播、focus/aria和export稳定态如何设计 | 把参考当静态PPT，不考虑HTML状态机 |
| 迁移结论 | 哪些经验可迁移到当前项目；哪些表层元素必须舍弃；哪些需要品牌色、真实素材、内容关系或交付格式改写 | 只列可借鉴点，没有不照搬项和改写依据 |

正向参考图深拆必须进入《参考经验转译卡》《页面族谱》《内容到UI/动效转译表》《动效脚本》《HTML状态机与交互合同》和QA字段。缺少这些落地路径时，正向参考只能标记为“已观察，未吸收”，不得进入Style Frame或正式生产。

## 通用型 / 专用型分类

视觉参考必须先进入以下两类之一。分类不是审美高低判断，而是使用边界判断。

### 通用型

适合在多种正式交付中复用的基础设计经验。通常可用于会议汇报、商务汇报、课程培训、客户提案、研究报告、内部复盘等场景，但仍需按项目语境调整。

典型特征：

- 信息层级清楚，风格不抢内容。
- 字体、色彩、网格、留白和组件具有稳定复用性。
- 视觉表达克制，能承载数据、流程、案例和结论。
- 不依赖强节日、强年龄、强娱乐或强品牌活动语境。

常见通用型方向：

| 方向 | 适用场景 | 注意事项 |
| :--- | :--- | :--- |
| 稳健商务 | 客户汇报、管理汇报、研究报告 | 避免变成通用蓝灰模板 |
| 高密度信息图表 | 研究报告、经营分析、复盘 | 必须保证结论锚点和可读性 |
| 编辑部杂志 | 专题报告、品牌故事、培训章节页 | 控制标题体量，避免装饰化 |
| 瑞士网格 | 框架说明、流程页、课程结构 | 避免只有网格没有叙事 |
| 品牌发布型 | 产品介绍、方案发布、路演 | 需要真实品牌资产支撑 |

### 专用型

只适合特定人群、特定节日、特定活动、特定品牌人格或特定内容题材的风格。专用型可以很优秀，但默认不得直接用于会议汇报、商务汇报、正式客户提案、常规课程培训和研究报告。

典型特征：

- 风格强于信息，容易改变内容气质。
- 依赖特定年龄、节日、活动、文化圈层或情绪场景。
- 视觉符号有强解释成本，正式场合可能显得不稳重。
- 大量使用手绘、童趣插画、节庆图形、潮流贴纸、漫画分镜、可爱拟物、强复古等表达。

常见专用型方向：

| 方向 | 适用场景 | 默认禁用场景 |
| :--- | :--- | :--- |
| 手绘 / 涂鸦 | 儿童、青少年、创意工作坊、节日活动、轻松破冰页 | 会议汇报、商务汇报、正式客户提案、严肃研究报告 |
| 节日主题 | 春节、周年庆、节日营销、员工活动 | 常规管理汇报、专业培训主视觉 |
| 童趣插画 | 低龄培训、亲子活动、教育启蒙 | 成人职业培训、审计/财务/合规主题 |
| 潮流贴纸 / 街头风 | 年轻消费品牌、社媒活动、校园招新 | 高管汇报、投融资路演、B2B方案 |
| 漫画分镜 | 故事化教学、服务流程演示、用户旅程戏剧化 | 数据报告、正式制度宣贯 |

### 分类决策规则

1. 若风格会明显改变项目的专业可信度，先归为专用型。
2. 若风格依赖特定人群或节日语境，先归为专用型。
3. 若风格只提供构图、留白、层级、图表处理等方法，可拆出通用经验；其表层视觉仍可归为专用型。
4. 专用型不是禁用型，但必须写明适用对象、适用页面、禁用场景和替代方案。
5. 正式商务或会议场景中使用专用型风格，必须在 Style Frame 记录中说明理由，并通过审美验收。

## 推荐记录格式

### Reference Entry

| 字段 | 要求 |

| 来源 | URL、截图路径、文件名或用户提供说明 |

| 权限状态 | 可访问 / 需登录 / 已截图 / 用户口述 |

| 关联规则 | 需要同步更新的 design standards、Page Spec、Style Frame 或 QA 项 |

### Design DNA 摘要

每组参考应尽量沉淀为以下三层：

1. Design System：颜色、字体、字号层级、网格、留白、组件、形状、阴影、图标、动效。
2. Design Style：情绪、叙事气质、构图方法、视觉重量、图像处理、品牌语气。
3. Visual Effects：背景效果、滚动/转场、图像动效、SVG/Canvas/WebGL、性能和降级策略。

## 团队生产中的使用方式

1. 项目启动时，先判断是否有适用参考条目。
2. 对候选参考先做通用型/专用型判断；专用型不得直接进入正式商务场景。
3. 做 Style Frame 前，从本库提取 1-3 个候选方向，而不是凭空生成“高级感”；需要风格检索或风格混合时，可读取 `references/visual/reference-entries.md` 中的 `REF-20260621-01`，把 StyleKit 作为设计库入口使用。
4. 填写《参考经验转译卡》：可借鉴经验、明确不照搬项、用户资料改写、品牌色与趋势取舍、页面族谱映射。
5. 做 Page Spec 时，把参考图中的版式关系转译为页面角色、核心观点、证据区和视觉锚点。
6. 做审美验收时，用本库中的禁用模式检查是否退回。
7. 项目结束后，把实际有效的设计经验回写到本库，失败经验也要保留。

## 参考库检索与经验回写门禁

完整视觉生产、Style Frame生成、审美退回修复、用户要求趋势/审美学习时，必须保留《参考库检索记录》。只说“参考过经验库”不算完成。

## 外部搜索接入规则

当用户要求联网搜索、当前趋势、最新审美、优秀网页案例，或当前参考库不足以支撑项目时，必须读取 `23-online-research-toolchain.md` 并执行外部搜索。

外部搜索结果不得停留在临时对话中，必须进入以下至少一项：

1. 《外部参考搜索记录》；
2. 《外部视觉参考转译卡》；
3. 新增或更新 Reference Entry；
4. 项目 evidence 中的《参考库检索记录》；
5. QA 记录中的参考落地证据。

只搜索不转译，不算使用参考。只转译不回写，不算经验沉淀。只采用不弃用，不算适配性审查。联网工具缺失时必须先按 `23-online-research-toolchain.md` 提请用户同意安装/启用；用户拒绝则停止当前联网参考任务，不得用本地参考库冒充外部近期验证。

| 字段 | 要求 |
| :--- | :--- |
| 检索时间 | YYYY-MM-DD HH:mm |
| 检索关键词 | 项目类型、受众、风格母线、页面角色、禁用风险 |
| 候选REF | 至少列出编号和采用/弃用判断 |
| 弃用REF理由 | 为什么不适合当前受众、内容密度、品牌色或交付场景 |
| 采用REF | 每个采用项必须说明可借鉴经验和不可照搬项 |
| 页面族谱映射 | 封面、问题、框架、数据、案例、收束等各自映射到哪些经验 |
| 趋势校准 | 是否需要外部近期验证；不需要时写明理由 |
| 责任副主编 | 品牌审美副主编和版式/UI副主编 |

视觉项目QA冻结前必须完成《经验回写记录》。成功经验和失败经验都要保留，避免团队只沉淀漂亮但不适配的方向。

用户点名的明显失败必须进入经验回写，不能只留在项目失败审计里。尤其是光学失衡、首态提前露出、语义阶段合并、动效生硬、页面丑、部门空转、工程烟测冒充QA等问题，都属于跨项目禁用经验。它们应被写入本库的失败经验、`04-design-standards.md`的设计硬失败、`17-motion-design-standards.md`的动效门禁或`07-qa-checklist.md`的QA规则。

经验回写不是总结一句“下次注意”。必须明确：这个失败会在哪些项目类型中复现、为什么会复现、哪些设计动作会触发、应该如何阻断、哪个reference已经更新。

### 参考证据门禁

“读过参考库”不等于“使用了参考库”。每个采用REF必须形成证据链：

| 字段 | 要求 |
| :--- | :--- |
| 已读取段落 | REF编号、读取的行号/段落标题、截图路径 |
| 提取规则 | 从该REF抽出的结构规则、色彩规则、组件规则、禁用规则 |
| 当前项目改写 | 哪些规则被改写为当前页面族谱、调色板、组件语法或动效脚本 |
| 落地证据 | 对应Page Spec页码、CSS变量/组件名/版式规则/截图路径 |
| 弃用证明 | 至少列出1个看似可用但被否决的REF或表层元素及原因 |
| 截图核验 | 截图中能看到哪些被采用的结构经验；哪些表层元素被刻意避免 |

缺少“提取规则 → 当前项目改写 → 落地证据 → 截图核验”任一环，不得宣称参考库真正使用。

## 团队 UI 设计条线与逻辑

本库的参考图不能按“好看风格”使用，而应进入一条稳定的设计生产链路：**场景判断 → 风格母线 → 品牌色与趋势校准 → 页面族谱 → 组件语法 → 动效/特效层 → 审美验收**。每一步都必须能回答“为什么这个风格适合这个交付对象”。

### 1. 先定交付场景，不先定风格

| 交付场景 | 默认视觉目标 | 优先参考方向 | 慎用/禁用方向 |
| :--- | :--- | :--- | :--- |
| 会议汇报 / 商务汇报 | 稳定、可信、结论先行、可投屏 | REF-20260619-05、08、17、21、34、36 | 童趣、手绘、最大主义、社媒黑底、过暗低对比控制台 |
| 课程培训 / 工作坊 | 清楚、节奏感强、步骤可跟随 | REF-20260619-05、07、09、15、17、32 | 高端黑金、金融收益感、过度实验字体、过多小字 |
| 研究报告 / 白皮书 | 证据优先、数据可信、版式耐读 | REF-20260619-08、11、17、21、36、42、43 | 纯广告海报、强娱乐色块、没有数据口径的伪仪表 |
| 产品发布 / 技术发布 | 明确主张、强记忆点、证据链完整 | REF-20260619-02、06、25、28、30、34、37、38、40、42、43 | 无真实产品/参数时禁用硬科技和遥测语汇 |

### 3. 品牌色优先，流行趋势辅助

参考图显示的 2026 设计趋势很明确：暗色任务控制、单一信号色、超大字体、硬网格、柔雾健康渐变、对象化品牌资产、高饱和彩块、黑白科研编辑风正在流行。但团队生产中必须遵守：**主品牌色优先，流行趋势只作为结构、强调和气质辅助**。

| 品牌/内容语义 | 推荐主色策略 | 可借鉴趋势 | 对应参考 |
| :--- | :--- | :--- | :--- |
| AI、开发者、基础设施、精密系统 | 黑/深蓝/深青 + 单一信号色：青、绿、黄或蓝 | 暗色控制台、点阵、等宽字、指标条 | REF-25、28、38、40、42、43 |
| 商业研究、咨询报告、经营分析 | 白/暖白/浅灰 + 红、绿、橙的语义强调 | 瑞士网格、横向条形图、结论页色带 | REF-05、08、17、36 |
| 健康、睡眠、情绪、专注 | 白/浅蓝/淡紫/粉橙柔雾 + 低饱和强调 | 柔和渐变、轻阴影、低压CTA | REF-21、22、23 |
| 年轻消费、活动、社媒传播 | 高饱和黄、粉、蓝、红 + 强口号 | 大字海报、彩块、贴纸、节奏化标签 | REF-09、14、18、24、27、33、37 |
| 会员、社区、身份、收藏 | 白/浅灰/蓝/黑 + 票券或徽章对象色 | 对象化身份、编号、邮票边、实体质感 | REF-26、39、41 |
| 文化、教育、儿童、工作坊 | 奶油底、橙绿蓝、低冲突色板 | 平面插画、课程卡、手册网格 | REF-07、15、16、32 |

禁用判断：若项目没有自己的品牌色，不能直接套“流行蓝紫渐变”或“黑底荧光绿”。先选择交付场景的安全母线，再从参考图提取一个有语义的锚点色。

每个项目至少要建立以下页面族谱。参考图只能帮助选择页面角色的视觉表达，不能用同一种卡片网格覆盖全稿。

### 5. 组件语法必须跟随风格母线

| 组件 | 瑞士信息系统线 | 编辑部杂志线 | 控制台/仪表系统线 |
| :--- | :--- | :--- | :--- |
| 标题 | 无衬线、大字号、强对比、常用左对齐 | 衬线/无衬线混排，标题像文章标题 | 无衬线或等宽，大写/短句，像系统状态 |
| 卡片 | 直角、hairline、少阴影或无阴影 | 可少量留白分组，不依赖卡片堆叠 | 暗色面板、细边框、状态角标 |
| 数据 | 条形、塔状KPI、横向对比 | 数据作为正文证据或图注 | 指标块、仪表、坐标、遥测、uptime |
| 图片 | 直角裁切，比例统一 | 图片是第一公民，允许大留白和图注 | 少用照片，多用系统图、网格、点阵、设备渲染 |
| CTA | 简短、直接、像系统动作 | 克制邀请式 | initialize、deploy、reserve、docs 等命令式 |
| 动效 | 数字计数、bar scale、SVG描线 | 淡入、图片轻微移动、章节翻页节奏 | 扫描线、状态点、曲线绘制、仪表锁定 |

### 6. Creative Production 路由

当参考库无法直接决定视觉方向时，调用 Creative Production 的思路，但不要把它当成“生成更多图”的捷径。它在团队流程里的角色是：先澄清创意意图，再生成可比较的视觉方向，最后才进入资产生产。

| 需求信号 | 路由 | 产出要求 |
| :--- | :--- | :--- |
| 不确定受众、卖点、场景 | Positioning | 明确受众、场合、增长目标、证明和禁用项 |
| 需要多种风格方向 | Mood boards | 形成 3-6 个视觉 territory，并映射到本库 reference |
| 需要产品/服务放进真实场景 | Scenes / Offers | 场景图或offer方向必须服务具体渠道 |
| 需要广告方向 | Ads | 产出广告构图、主张、格式和风险边界 |
| 已有一张参考图但要变体 | Shots | 只变构图、视角、裁切或焦点，不重写品牌逻辑 |
| 品牌标志或系统未定 | Logos | 先解决识别系统，再进入页面视觉 |
| 已选定方向要出图 | Assets / Generative Polish | 只在方向确定后生产发布资产 |

### 9. 使用本库生成 Style Frame 的最小流程

1. 写清楚交付场景、受众、可信度要求和观看距离。
2. 从“风格谱系索引”选 1 个主方向和最多 1 个辅助方向。
3. 先写当前用户资料锚点：品牌色、Logo/截图/案例/数据、受众、使用方式、禁用事项。
4. 选择主品牌色；若无品牌色，从场景语义中选单一锚点色，不从流行图里随机抄色。
5. 填写参考经验转译：借鉴结构和组件语法，不复制参考图的行业语汇、文案、装饰和无关色彩。
6. 定义页面族谱：封面、问题、框架、证据、案例/截图、数据、路线图、收束。
7. 为每个页面族谱指定一个参考条目，不允许全稿只套一个参考。
8. 输出 Style Frame 时必须包含：字体层级、色彩系统、版心、组件、页眉页脚、关键视觉锚点、禁用项。
9. 进入 HTML/PPT 生产时，再映射到 guizang 或 html-ppt 的主题和布局。
10. 审美验收时按 `04-design-standards.md` 和 `11-aesthetic-anti-ai.md` 检查：是否品牌化、是否避免AI模板、是否每页都有真实信息承载、是否没有照搬参考图表层。

## 外部资料获取顺序

当视觉参考来自私有云文档、网页、共享白板或其他外部资料时，按以下顺序获取：

1. 直接读取页面内容和图片资源。
2. 若页面支持导出，优先导出图片包、PDF或HTML；私有或版权不明资料只能进入项目本地 evidence，不得进入公开Skill仓库。
3. 若无法下载但页面可见，使用截图读取：按屏或按图片组截图，保存到项目本地 evidence，再逐组分析；公开版本只保留脱敏后的抽象经验。
4. 若页面需要登录或权限确认，先请用户在浏览器中完成登录/授权；不代输账号、密码或验证码。
5. 若只能获得低清截图，先做风格级判断，不做像素级色值或字号结论。
6. 不管来源方式如何，最终都必须沉淀为 Reference Entry 和 Design DNA 摘要。

截图读取时不得逐图写流水账，应按风格簇整理：先归类，再提炼通用型/专用型边界、可复用经验、禁用场景和生产规则。若外部文档中的图片可点开大图，必须点开大图后截图；画册缩略图只能用于定位，不得作为视觉分析依据。
