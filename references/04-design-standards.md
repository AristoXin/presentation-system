# 04-design-standards

读取时机：当任务涉及视觉系统、版式、表格页、正文色阶、中英混排、页面族谱、问题页结构修复或设计质量审查时读取。若涉及“AI味/审美/高级感/Style Frame”，同时读取`11-aesthetic-anti-ai.md`。若用户提供视觉参考图、设计风格链接或灵感板，同时读取`15-visual-reference-library.md`。若涉及design-dna、PPT-DNA、guizang、HTML/PPT/Slides视觉生产或用户质疑审美/UI/动效没有深度集成，同时读取`20-design-dna-guizang-integration.md`。若用户指出排版/UI不好看、layout重复、卡片感重、stylekit/taste等Skill未集成，同时读取`21-design-skill-stack-integration.md`。

## 视觉标准

- 品牌色优先，流行趋势只做辅助。
- 页面必须有清晰的信息层级：主标题、主观点、证据、辅助说明、页脚。
- 每页必须有一个明确视觉重心，不得让多个模块平均分布。
- 留白必须稳定，版心必须统一。
- 色彩必须克制，优先使用1个主色、1个辅助色、1个风险/强调色。
- 卡片、表格、标签、进度条等组件必须统一，不得像不同组件库拼接。
- 不同页面应有不同叙事结构，不得复制同一版式覆盖全稿。
- 视觉设计必须服务当前用途：客户汇报、培训、路演、研究报告、复盘等不能共用同一种模板。
- 设计应有人工设计痕迹：克制、取舍、对齐、节奏、层级、细节，而不是生成式堆叠。
- 页面必须有明确视觉层级，不用装饰掩盖内容空洞。
- 表格页优先通栏或宽栏，必须有结论锚点。
- 正文色阶保持稳定，章节变化不能造成阅读断层。
- 中英混排采用组件化规范，英文术语不得抢走中文主叙事。
- 点名问题页必须重写Page Spec，不以微调边距作为主要修复手段。
- 完整页面生产前必须通过Style Frame审美闸门，详见`11-aesthetic-anti-ai.md`。
- 页面不得默认使用通用AI模板式蓝紫渐变、玻璃拟态、全页卡片网格、胶囊标签泛滥和伪图表。
- 外部视觉参考必须先转译为场景化规则，再用于页面生产；不得把参考图的表层装饰直接搬进项目。
- 每次设计必须证明当前方案来自用户资料、品牌色、受众场景和内容证据；参考图只能提供经验，不提供可复制页面。
- 主品牌色优先于参考图色彩。除非用户明确要求重塑品牌，否则不得用参考图的流行配色替换项目主品牌色。
- 风格必须先判断通用型/专用型；手绘、童趣、节日、潮流贴纸、漫画等专用型风格不得默认用于会议汇报、商务汇报、正式客户提案、常规课程培训和严肃研究报告。
- 文本与背景必须有明确对比。黑底黑字、深底深字、浅底浅字、图片上无遮罩文字等低对比情况，一律退回版式/UI链路。
- 线条、边框和分隔符必须有统一语义。不得在同一页面或同一组件族中无解释地混用黑、红、绿等高冲突线条。
- 表格必须建立独立组件规范：表头显著高于正文，正文按字段类型统一对齐，不得同一列或同类字段忽左忽右。
- 缩写、模型、法则类内容必须建立“记忆组件”规范。字母、英文全称、中文含义和解释句必须形成清晰绑定，不得把首字母和英文全称拆散。
- 类表格、证据清单、判断清单必须统一对齐。不能同一组件中有的左对齐、有的右对齐；完整短句不应因列宽不当被拆成破坏语义的两行。
- UI结构必须来自内容关系。版式/UI副主编必须读取语言内容副主编的《内容到UI/动效转译表》，再决定通栏、对比、流程、证据旁注、工作台、表格、路线图等结构。没有内容转译表时，不得用卡片矩阵、左右栏或表格模板直接承载页面。
- UI设计不得被降级为“对齐几个方块、放一套配色、填入几段文字”。正式PPT/Slides设计必须同时说明跨页视觉语法、页面族谱、信息层级、真实视觉锚点、组件角色、动效/静止策略和交付路线；缺任一项时，不得标记为UI方向已确认。
- 正向参考图不得被降级为“好看所以照着做”。每张被采用的正向参考都必须拆出配色原因、文字协同、字体系统、版面规划、页面切换、替代方案、动效方案、交互/HTML状态机和迁移边界；缺少《正向参考图深拆与经验总结表》时，不得进入UI方向确认。

## Template-Like Layout Is a Design Failure

美观不等于套用统一模板。

以下情况必须触发 Anti-Template Review：

- 连续 3 页使用同一布局族。
- 超过 50% 页面为标题 + 卡片。
- 超过 50% 页面为左右栏。
- 所有页面使用相同背景、相同卡片、相同动效。
- 页面结构没有根据内容关系变化。
- 每页都像独立信息卡，而不是叙事系统的一部分。

设计部门必须证明页面族谱来自内容关系，而不是来自模板惯性。

## 布局强制置换矩阵

当 Page Spec 或 Style Frame 出现以下默认模板倾向时，版式/UI副主编必须先尝试右侧置换布局，并记录采用、改写或弃用理由。连续两页不得直接复用同一默认模板；全稿超过 40% 页面落在同一布局母版时，必须退回页面族谱。

| 默认模板布局（高风险） | 优先置换布局 | 适用页型 |
| :--- | :--- | :--- |
| 3列等宽卡片 + 图标 | 对角线对比布局：左上角超大主判断，右下角表格、截图或证据形成斜向张力 | 数据对比、产品卖点、风险/收益 |
| 居中标题 + 下方三点列表 | 侧边栏沉浸布局：左侧 30% 固定目录/判断，右侧 70% 承载流式内容和证据 | 章节过渡、流程总览、方法框架 |
| 顶部标题 + 底部双栏正文 | 杂志跨页布局：一侧真实图片/图形/证据承重，另一侧以倾斜或错位文本区承载解释 | 核心观点、案例转折、收束行动 |
| 竖向 bullet list | 横向路径或蜿蜒时间轴：要点转为节点、状态点、证据点和讲述动作 | 步骤讲解、机制演变、项目计划 |
| 大色块标题页 | 证据先行章节页：用真实截图、数据片段或引用作为第一眼锚点，标题让位于判断 | 章节开场、失败审计、研究发现 |

执行检查：

- UI 初审必须统计每页布局母版和相邻页重复情况。
- 若连续 2 页使用同一默认模板且没有内容关系理由，退回 Page Spec。
- 若置换布局会牺牲品牌规范、可读性或交付格式，可弃用，但必须在 Anti-Template Review 中写明理由。

## 多页视觉系统门禁

当用户提供的参考图、截图或样张中包含多页PPT内容时，必须先拆页分析，再抽象规则。不得把长图当作单张海报参考，也不得只抽取配色、字体或装饰。

| 维度 | 必须回答 | 硬失败条件 |
| :--- | :--- | :--- |
| 页面族谱 | 每一页分别承担封面、洞察、策略、步骤、Demo、案例、收束中的哪种角色 | 只说“整体风格好看”，没有拆页角色 |
| 跨页视觉语法 | 哪些网格、标题容器、色带、边框、图像槽、页眉页脚形成连续系统 | 每页重新摆方块，或只复用同一个模板 |
| 信息层级 | 第一眼主判断、第二层证据、第三层注释/口播/标签如何分工 | 大标题、正文、标签、装饰同权重 |
| 真实视觉锚点 | 人物、商品、截图、图表、关键数字或结构图如何承重 | 只有抽象色块、图标和占位框 |
| 组件语义 | 卡片、引用、箭头、便签、撕纸、网格、斜切色块分别承担什么信息角色 | 装饰多于信息，组件无法说明为什么存在 |
| 动效协同 | 哪些内容初始可见，哪些由讲者触发，哪些需要静止，哪些适合HTML/PPTX/图片PPT | 全页统一淡入，或分步内容提前露出 |

正向参考图深拆必须额外回答：为什么这样配色；文字如何配合画面；字体、大小、字重和样式为什么成立；版面如何规划视觉重心和阅读路径；页面之间如何切换；如果当前团队重做，会提出哪几种方案；动效与交互如何服务HTML产物。若只复述参考图“视觉冲击强、配色鲜明、版式活泼”，视为未拆解。

若采用 `REF-20260622-01` 这类社媒营销 Campaign Deck 参考，UI副主编必须把“高饱和拼贴皮肤”和“跨页设计方法”分开：前者默认专用，后者可抽象为页面族谱、信息植入、真实素材锚点和分步动作。品牌审美、语言内容、动效设计、交互工程和QA必须共同签署转译结果。

## StyleKit审美组件转译规则

StyleKit不是单纯的提示词工具，也不是最终审美答案。它在本系统中作为审美组件库使用：品牌审美副主编负责提取审美意图，版式/UI副主编负责提取token和组件架构，动效设计副主编负责提取state/interaction的演示语境转译，QA副主编负责验证这些规则是否真实落地。

使用StyleKit或`REF-20260621-01`时，必须完成以下转译：

| 维度 | 演示文稿转译要求 | 硬失败条件 |
| :--- | :--- | :--- |
| Intent First | 记录目的、受众、语气、记忆钩子，并说明它如何影响封面、关键页和收束页 | 只写风格slug或形容词，没有当前项目意图 |
| 反泛化 | 执行swap、squint、signature、token四项测试，确认页面不是可互换模板 | 任一关键页换成其他项目也成立，或3秒内没有独特主视觉/主判断 |
| Token层级 | 建立Brand → Semantic → Component → State的颜色、字阶、间距和边框语义 | 同一颜色/字阶/间距在不同页承担冲突角色 |
| 组件架构 | 把Base → Variant → Size → State → Override转译为卡片、表格、标注、引用、进度、控件等组件规则 | 组件只靠视觉相似，没有variant、size、state和override边界 |
| 所有权分配 | 每个色彩、字体、间距、动效决策必须只有一个来源：品牌、内容关系、参考经验、StyleKit或交付约束 | 多个参考/风格混用，无法解释谁拥有某个视觉决定 |
| 状态转译 | frontend的hover/focus/disabled/reduced-motion等状态转为演示的首态、揭示态、强调态、export稳定态或不适用说明 | 把网页交互状态机械搬进PPT/HTML slide，或缺少export稳定态 |
| 可访问性 | 普通文本对比度、重点文字可读性、交互HTML的focus-visible、触控目标、reduced motion必须被检查 | 为风格牺牲投屏阅读、焦点状态、触控可用性或动效舒适度 |

StyleKit的`hard_prompt`、`style_choice`、slug或前端组件词不得直接进入最终Page Spec。它们必须先变成《参考经验转译卡》《页面族谱》《UI方向卡》《动效脚本》或《审美QA记录》中的可检查字段。若只记录“已读StyleKit/采用某slug”，但没有上述转译，结论必须写“已读取未吸收”，不得标记审美通过。

## PPT-DNA与guizang深度设计规则

执行合同以`20-design-dna-guizang-integration.md`为准。本节只定义设计标准；正式视觉生产必须按`20`输出Presentation Design DNA档案、DNA来源追踪表、guizang桥接表、UI吸收表、动效吸收表和QA吸收检查。

PPT-DNA和guizang必须服务“用户资料如何长成当前演示设计”这个问题。PPT-DNA负责抽取设计身份，guizang负责在演示版式、主题节奏、图片槽位和动效中落地。二者都不能先于内容关系、受众场景、页数充分性和页面族谱成为设计答案。

| 能力 | 品牌审美要求 | 版式/UI要求 | 硬失败条件 |
| :--- | :--- | :--- | :--- |
| PPT-DNA design_system | 从用户品牌、参考图和交付场景定义颜色、字体、间距、布局、形状、组件、动效token | 将token转成调色板语义表、字阶、网格、组件族、表格/卡片/标注规范 | 只写“设计系统/高级感”，没有可追溯token |
| PPT-DNA design_style | 提取mood、genre、personality、visual language、composition、brand voice | 将composition转成页面族谱、视觉重心、留白策略、阅读路径 | 只有形容词，没有页面级构图规则 |
| PPT-DNA visual_effects | 判断WebGL、Canvas、SVG、动效是否符合受众、设备、导出和观看距离 | 将效果转成HTML实现、PPT/PDF静态降级、截图策略和低功耗策略 | 为炫技加入效果，导致可读性、导出或性能风险 |
| guizang风格适配 | 在A电子杂志/B瑞士国际主义之间做适配判断，说明为什么适合当前资料和场景 | 先列叙事弧、页数规划、主题节奏，再建立逐页layout/Sxx映射 | 先选瑞士风、杂志风、IKB或模板，再倒推内容 |
| guizang版式映射 | 审查A/B风格是否与品牌气质、内容密度、素材类型冲突 | 建立页码→页面角色→内容关系→layout/Sxx→选用理由→图片槽位→动效意图 | 无逐页映射、无改造差异，或自造未登记瑞士版式 |
| guizang主题节奏 | 判断hero/light/dark节奏是否服务讲述张力和观看舒适度 | 检查连续同主题、单一accent、中文标题分档、投屏最小字号、底部安全区 | 连续模板页疲劳、字号过小、图片比例混乱、内容碰到底部控件 |

若采用PPT-DNA，只输出Design DNA JSON不算设计完成；必须落到页面族谱、UI方向、组件规范、动效脚本和QA字段。若采用guizang，只说“用瑞士风/杂志风/guizang模板”不算设计完成；必须有风格适配理由、主题节奏表、逐页版式映射、模板改造差异和截图核验。缺少这些字段时，分别标记为“已提取未吸收”或“已选模板未吸收”，不得进入全稿生产或审美通过。

## 视觉舒适门禁

视觉舒适不是主观“好看”，必须用截图和页面结构共同判断。

| 检查项 | 硬失败条件 |
| :--- | :--- |
| 调色板语义 | 黑/白/蓝或多强调色没有角色说明；同一颜色在不同页承担冲突语义 |
| 色彩数量 | 正式汇报/培训页面超过1个主accent且无业务分类理由 |
| 页面重心 | 同一页出现3个以上同权重模块争抢注意力 |
| 文字密度 | 标题、正文、注释、标签全部大号/加粗，无法形成主次 |
| 行长与行距 | 正文长句被压成拥挤块，或短句被列宽拆成破坏语义的随机换行 |
| 留白节奏 | 页面边缘、模块间距、标题到正文距离明显不成体系 |
| 背景负担 | 背景、网格、纹理、色块比内容更抢眼 |
| 观看判断 | 封面3秒内看不出主题；内容页3秒内看不出主判断；练习/工作台页10秒内不知道操作点 |

任一硬失败出现时，UI副主编必须退回Page Spec或Style Frame。禁止仅通过改边距、换颜色、降低透明度来绕过。

## 光学平衡门禁

版式通过不等于元素对齐。UI副主编和品牌审美副主编必须同时检查“光学重量”：页面上半区/下半区、左半区/右半区、主视觉/辅助信息、标题/正文、留白/内容块是否形成稳定张力。

| 检查项 | 硬失败条件 | 处理 |
| :--- | :--- | :--- |
| 上下重量 | 头轻脚重、头重脚轻、底部堆积或顶部空洞导致主题不稳 | 退回Page Spec，重新定义主视觉、标题区和证据区 |
| 左右重量 | 左重右轻、右重左轻、单侧文字/卡片/深色块过密，另一侧缺少承重 | 退回版式策略，重分栏宽、视觉锚点和留白 |
| 阅读路径 | 第一眼落点不是主判断，视线被页脚、装饰、侧栏或次要卡片吸走 | 重写视觉层级，不许只调边距 |
| 视觉锚点 | 页面缺少真实锚点，靠大色块、卡片或装饰承重 | 补真实素材/结构图/关键数字，或降级为Page Spec |
| 跨页节奏 | 相邻关键页出现相反但同样失衡的重心漂移 | 回到页面族谱，重排页组节奏 |

封面、章节页、核心模型页、案例页、工作台页、收束页必须填写光学平衡结论，并附标注截图或三分区记录路径。证据至少说明上半区/下半区重量、左半区/右半区重量、第一眼落点、主视觉路径和需要联动检查的同类页。用户点名某页失衡时，该页同类页面必须联动检查。出现上述硬失败时，任何“条件通过”“基本通过”“工程烟测通过”均无效。

## 跨项目硬失败经验

以下问题已经被用户明确指出为“很明显的问题”，不是某个项目的局部偏好。任何后续视觉项目、HTML演示、PPT、PDF或截图包中出现同类问题，都必须按通用设计失败处理，不能解释为“当前项目风格选择”。

| 失败信号 | 跨项目判定 | 必须动作 |
| :--- | :--- | :--- |
| 封面、章节页或核心页头轻脚重、左重右轻、右重左轻 | 光学平衡失败 | 重写Page Spec；重新安排标题、主视觉、辅助信息和留白，不许只调边距 |
| 相邻关键页重心相反且都不稳 | 页面族谱失败 | 回到页组节奏，重排封面、模型页、案例页、收束页的明度、密度和主视觉节奏 |
| 模型页/清单页首态已露出第1点或答案 | 分步设计失败 | 转交`17-motion-design-standards.md`，补首态/后续态矩阵并重做状态 |
| 流程页把不同语义阶段合并显示 | 内容到UI转译失败 | 拆成独立阶段、分步态、子页或讲者备注；不得用表格硬塞 |
| 页面被用户评价为丑，且不是单个小组件问题 | 视觉母线失败 | 重写视觉母线、页面族谱和Page Spec；不得只改CSS变量、配色或组件圆角 |
| 部门记录显示通过但用户可见问题仍存在 | 协作治理失败 | 复核记录无效，必须重新进入对应副主编复审和QA回归 |

这些失败信号必须写入项目《上一版问题关闭矩阵》和《经验回写记录》。没有关闭矩阵时，不得生成新版全稿；没有经验回写记录时，不得把项目经验称为已沉淀。

同一页面若同时使用黑、白、蓝或其他强识别色，必须先填写本表。没有语义表，不得标记审美通过。

| 颜色 | 角色 | 允许页面/组件 | 禁止用途 | 截图核验证据 |
| :--- | :--- | :--- | :--- | :--- |
|  | 背景/正文/弱文本/主品牌/唯一accent/风险/状态/分隔 |  |  |  |

正式汇报和培训类页面默认只允许一个主accent；第二accent必须有业务分类理由。黑白蓝同时承担多个无关角色、同一颜色跨页角色漂移、或为了“有设计感”随机换色，均为QA否决项。

## UI aesthetic gate

生成任何HTML slide、PPTX、PDF视觉稿或截图包前，必须通过UI aesthetic gate。该gate不只检查是否整齐，而是判断页面是否具备专业设计感。

若任务涉及排版/UI不好看或审美退回，UI aesthetic gate必须增加`21-design-skill-stack-integration.md`中的TasteSkill吸收合同：Design Read、DESIGN_VARIANCE、VISUAL_DENSITY、Section/Layout Repetition Ban、卡片/形状/颜色纪律和pre-flight结论。缺少该吸收合同，不得进入Style Frame或全稿生产。

模板不可先行：任何模板、版式库、主题色、历史样张或主设计Skill都不得先于内容分析成为设计方向。UI副主编必须先读取语言内容副主编的《内容到UI/动效转译表》、品牌审美副主编的《参考经验转译卡》和动效设计副主编的动效方向，再定义页面族谱。模板只允许作为实现语法，用来承接已经定好的页面角色、视觉锚点和讲述动作；若页面族谱来自模板可用版式，而不是来自当前材料的内容关系，直接退回。

模板适配性审查：用户提供内容后，第一判断不是“能套哪个模板”，而是“这份内容要求什么结构”。候选模板必须证明适合当前信息关系、受众观看距离、讲述/训练动作、证据强度、品牌语气和动效节奏；不适合时必须弃用。为了套模板而删减训练动作、弱化证据、平均化页面权重、把案例页做成通用卡片页，均属于设计失败。

模板改造规则：模板通过适配性审查后，UI副主编必须把它拆成可修改的结构，而不是照搬。每页至少回答：本页保留模板的哪一层骨架，哪些区域必须为当前内容重写，哪些组件需要新增或拆分，哪些模板装饰会干扰信息必须删除，文本密度如何调，视觉锚点如何替换为当前资料，动效如何服务讲述动作。只替换文案、改主题色、搬原模板布局，不算改造。

禁用项：

1. 禁止默认使用大面积渐变、玻璃拟态、随机网格背景、无意义斜切色块。
2. 禁止每页都使用“标题 + 多张卡片 + 图标”的机械结构。
3. 禁止色彩过多、阴影过重、装饰线过多。
4. 禁止模块之间视觉权重平均，必须有明确主视觉、主结论和阅读路径。
5. 禁止把内容简单塞入卡片矩阵，必须根据内容关系设计结构。
6. 禁止用“通用科技感”“通用商务感”“通用咨询风”替代针对项目的视觉判断。
7. 禁止为了显得丰富而堆叠图标、标签、徽章、色块。
8. 禁止文字层级混乱：标题很大但正文、标签、注释缺少清晰层级。
9. 禁止背景纹理喧宾夺主。
10. 禁止页面一眼看上去像AI模板库作品。
11. 禁止黑底黑字、深底深字、浅底浅字等低对比组合。
12. 禁止无语义的多色线条和边框混用；红/绿/黑等强语义颜色只能用于明确状态或风险分类。
13. 禁止表格表头和正文缺少视觉差异，或表格正文对齐规则混乱。
14. 禁止缩写模型组件将首字母、英文全称、中文解释错误拆分，导致记忆关系断裂。
15. 禁止同类列表/表格中短句被不合理换行，或同类字段左右对齐混乱。
16. 禁止用“采用瑞士风/杂志风/某主题色/某模板编号”替代审美判断。
17. 禁止把每次不同用户资料塞进同一套母版；历史样张只能提供失败经验或局部组件经验，不能作为新版默认结构。
18. 禁止候选模板不适合当前内容时仍硬套；不得为了模板整齐牺牲用户资料中的证据、训练动作、案例冲突或讲述节奏。
19. 禁止模板适合大方向后直接调用；必须经过逐页改造差异设计，不能只替换文案和颜色。
20. 禁止黑、白、蓝或其他强识别色无语义混用。没有《调色板语义表》时，不得进入审美通过。
21. 禁止只通过改CSS变量、换色或调边距回应用户审美退回；必须先判断是否需要重写视觉母线、Page Spec和页面族谱。

必须主动检查项：

1. 视觉系统是否来自项目语境，而不是只用“深蓝 + 米白 + 青橙强调色”这类通用商务安全解。
2. 候选风格是否已经判断为通用型或专用型；若是专用型，是否存在明确适用对象、使用页面和禁用场景。
3. 是否有《参考经验转译卡》，说明参考图的哪些结构经验被采用、哪些表层元素被排除、当前用户资料如何改写方案。
3b. 若采用正向参考图，是否有《正向参考图深拆与经验总结表》，逐项回答配色、文字、字体、版面、跨页切换、替代方案、动效、交互和迁移边界。
3a. 是否有《内容到UI/动效转译表》，说明每页内容关系如何决定UI结构、视觉锚点和动效/静止策略。
4. 主品牌色、用户素材和业务证据是否优先于参考图流行色；若借鉴趋势色，是否只作为辅助色、强调色或氛围层。
5. 页面族谱是否真的变化。连续3页以上都依赖“标题 + 等宽卡片/面板/网格”时，必须重写至少一半页面的Page Spec。
6. 关键页是否有真实视觉锚点。案例、系统、品牌、数据页不能只靠虚线占位框或文字说明承重。
7. 字号是否服务观看距离。1920x1080培训或汇报页中，普通正文通常保持22-30px；若大量正文超过32px或加粗到900，必须说明投屏距离和删减依据。
8. 标题是否压迫正文。标题不应因为“大”而显得有设计，长标题应优先压缩文案、换行或降级字号。
9. 色彩语义是否稳定。超过3个强调色必须有明确分类含义；不能为了避免单调而给同类卡片随机换色。
10. 背景纹理、斜切色块、边框和阴影是否承载信息。只负责“看起来有设计”的装饰应删除。
11. 对比度是否足以投屏阅读。标题、正文、表格、页脚和图注不得出现与背景相近的颜色。
12. 线条颜色是否服务信息分类。同一组件族的线条颜色、粗细、透明度必须统一；风险色不能被当装饰色使用。
13. 表格表头是否用字重、背景、分隔或留白建立层级；正文数字、文本、标签列是否按统一规则对齐。
14. 缩写/术语组件是否有稳定语法，例如 `SITUATION` 中首字母可放大、虚化或强调，但必须与 `ituation` 同组；中文“情境”作为解释层，不得插在 `S` 和 `Situation` 之间破坏映射。
15. 类表格组件是否通过列宽、字阶、行高保证短句完整，不靠随意换行消化空间。
16. 页面族谱是否由当前内容生成：讲授页、案例页、带练页、工具页、AI复核页、收束页是否各有不同的视觉任务，而不是同一模板改标题。
17. 设计引擎选择是否后置：是否先完成部门一致意见和内容到UI/动效转译，再选择guizang/html-ppt/frontend-slides等Skill承接。
18. 参考库和趋势调研是否真的发生了否决：是否列出不适合当前项目的模板、趋势和历史样张，并说明弃用理由。
19. 模板改造是否真的发生：是否有逐页保留/改写/新增/删除/调整记录，且调整来自当前内容。

## 页面族谱

封面页、目录页、章节页、概念页、流程页、表格页、案例页、演练页、收束页必须分别建立版式规则。

## 结构优先

页面不好看时，先检查核心观点、证据层级、视觉锚点和Page Spec，再调整颜色、阴影或边距。

## 当前HTML类问题修复指导

当处理已有HTML演示且用户反馈“AI味重/不好看/控件碍眼”时，优先执行：

1. 先做HTML审美审计：列出页数、页面族谱、重复版式、卡片/面板密度、文字密度、占位素材、控制条状态和截图可用性。
2. 重构视觉系统，减少网格背景、斜切色块、重复卡片和高饱和色。
3. 提高页面留白质量和标题层级；长标题先改文案，再调CSS。
4. 让每页版式根据内容关系变化，而不是机械套两栏。
5. 对连续卡片页做结构替换：对比页用冲突结构，流程页用路径结构，案例页用证据原件/判断旁注结构，行动页用清单和承诺区结构。
6. 对关键占位页做降级判断：缺Logo、截图、简历、系统界面时，不得用大虚线框当正式视觉锚点；要么接入真实素材，要么改为结构化证据卡并在QA中标为未冻结。
7. 若底部`.deck-controls`默认强显示，转交`06-interaction-standards.md`的Presentation Chrome规则处理。
8. 修完后用截图检查每页是否仍像AI模板；若无法截图，必须手工浏览器逐页复核并把截图缺失列为冻结阻断项。
9. 若仍像AI模板，继续改Style Frame或Page Spec，不只微调CSS。

## 表格页规则

1. 列数超过3列或正文超过2行时，优先通栏或宽栏。
2. 表格页必须有结论锚点，不只展示表格。
3. 表头、行高、列宽、斑马纹、重点列和注释规则必须统一。
4. 表格正文色阶与普通正文一致。
5. 表头必须比正文显眼，至少通过字重、背景、色阶、边界或留白中的两项建立层级。
6. 正文对齐必须按字段类型统一：文本/描述通常左对齐，数字/金额/百分比通常右对齐或小数点对齐，状态/标签可居中；同一列不得混用。
7. 表格线条优先使用低饱和中性色或品牌辅助色；红、绿等风险色只能用于明确状态，不得作为普通分隔线。
8. 表格所在页不得同时出现多套边框颜色、行分隔颜色和强调线规则。
9. 类表格组件也适用本规则，包括证据列表、候选人判断行、风险清单、STAR/模型清单、行动清单。
10. 同一列必须使用同一对齐方式。状态列可居中或右对齐，但同一组件不得混用。
11. 短判断句优先通过加宽列、缩短文案、降低字号或改变组件结构保持完整；不得把“遇到困难不主动求助”这类完整判断拆成破坏阅读的随机两行。

## 缩写与术语组件规则

1. 缩写法则、模型、英文术语必须由语言内容副主编和版式/UI副主编共同定义组件，不得由执行层拼接字段。
2. 字母与英文全称必须绑定。可用方式包括：大写首字母虚化 + 余字清晰、`S / Situation`、`SITUATION` + 中文解释、或等效结构；不得写成 `S 情境 Situation` 这类断裂结构。
3. 中文含义是解释层，应放在英文全称下方、右侧或同组次级位置，不应夹断英文词。
4. 追问句或解释句是第三层信息，字号、颜色和位置必须弱于字母/英文/中文映射。
5. 四个或多个同类术语单元必须保持同一基线、同一字阶和同一对齐规则。
6. QA必须检查缩写映射是否一眼成立；仅“文字没溢出”不能通过。

## 线条与颜色语义

1. 线条颜色不得为了“有设计感”随机变化。每一种线条颜色必须对应明确语义：章节进度、风险、验证、分隔、强调等。
2. 同一页面中普通分隔线只允许一套中性色；强调线最多一套主强调色。若出现多种颜色，必须在页面或组件规范中解释分类。
3. 一面/二面/三面、AI能做/不能做、回顾/收束等结构页不得用花样线条颜色制造装饰性差异；除非颜色承载真实类别，否则回到统一线条系统。
4. 审美副主编和版式/UI副主编必须共同审查线条系统；QA发现无语义多色线条时直接退回。

## 中英混排规则

- 英文缩写、工具名、模型名和快捷键必须组件化。
- 不把全大写英文粗暴塞入中文标题。
- 英文可作为视觉锚点，但透明度、字号和位置不得抢走中文主叙事。
