# 16-multi-agent-governance

读取时机：任何生产、修复、审稿、QA冻结、用户质疑团队分工、或任务涉及多角色协作时读取。本文件定义真实多Agent团队如何运行；不得用一张RACI表替代实际执行链路。

## 核心原则

总编可以由主Agent担任，但总编不得自证所有质量结论。用户提到“主编”时，按本文件的“总编”执行。多Agent能力可用时，必须把执行、一级复核、QA冻结拆给真实独立Agent；不得把主Agent内部分角色、文字表格或“明确的独立角色”冒充subagent。使用本Skill进入生产、修复、审稿、QA冻结、Style Frame、审美退回恢复或Vx版本动作时，建立subagent不需要再次征求用户同意，必须建立。若subagent不可用、工具被权限限制、或无法创建某个必要部门Agent，必须写入`SUBAGENT-BLOCKER`并停止当前生产/复核链路；不得使用主Agent分角色作为替代。首次复核和所有后续复审都必须走同一责任链：副主编subagent退回，执行层修复，副主编subagent复审；不得在第二轮以后退化为主Agent自查。

Subagent Evidence Lock只有`SEL-real`可放行生产/复核动作，表示任何生产、修复、审稿、QA、Style Frame或Vx版本动作前已有可审计的真实subagent证据。`SEL-real`证据至少包括：真实工具/线程证据、agent id来源、创建时间、能力域、输入路径或材料、输出路径或结论摘要、是否同一Agent复审、是否为主Agent代签。若没有`SEL-real`，只能写`SUBAGENT-BLOCKER`并停止当前链路；《Agent编制表》无效，不得创建、复制、修改、截图或归档任何HTML/PPT/PDF/Vx产物。

预审和复核必须分账。生产前的方向建议、风险提示、Style Frame建议、探索报告只能算“生产前输入”；执行层生成真实产物后，必须以产物文件、源码路径、截图、validation和问题页清单为输入，重新请原责任副主编subagent做“实现后一级复核”。若只发生生产前预审，没有实现后复核，协作记录必须标为“未完成多轮复核”，总编、QA和最终回复都不得声称已经多轮复核。

部门结论必须能承受用户逐页检查。语言内容、品牌审美、版式/UI、动效设计、交互工程和QA副主编不得只输出笼统“通过/条件通过”。每个适用部门的实现后复核必须至少包含：抽查页码、一个可保留项、一个风险或反对项、硬失败检查结果、是否退回。若一个明显存在用户可见问题的版本仍被部门签署“条件通过”，该签署无效，并触发失败审计；总编不得把无效签署汇总为团队通过。

## 默认建立与工具约束

本Skill把 subagent 视为正式生产和复核的默认组织能力，不再向用户询问是否同意创建。以下用户表达只用于强化优先级，不是创建前提：

- “subagent / 子agent / 多Agent / 并行Agent / agent swarm”
- “副主编真的参与了吗”
- “不要主Agent自审”
- “副主编深度参与”
- “团队分工”
- “团队分工是否真实执行”
- “审美/UI/QA部门有没有深度参与”
- “你已经多次被要求调用子Agent”

凡本Skill要求多Agent，主Agent必须创建真实能力域Agent，不能以“用户未明确要求subagent”为理由降级。若工具不可用，必须停止生产，写入`SUBAGENT-BLOCKER`，并向用户说明无法满足正式团队流程；不得继续生成HTML/PPT/PDF并事后补表。

## 三级团队

| 层级 | 角色 | Agent实现 | 核心产出 |
| :--- | :--- | :--- | :--- |
| 决策层 | 总编 | 主Agent | 总编目标、取舍、最终二级复核、是否进入QA |
| 决策层 | 副总编 | 独立审查Agent；不可用时必须停止当前链路并写`SUBAGENT-BLOCKER` | 部门一致性审核、冲突协调、新部门申请预审、是否可交总编 |
| 管理层 | 能力域副主编 | 独立审查Agent；不可用时不得直接由主Agent代替，必须停止当前链路并写`SUBAGENT-BLOCKER` | 一级复核：通过/退回/条件通过 |
| 执行层 | 专业专员 | 独立生产Agent或主Agent执行 | 页面、代码、截图、问题修复、测试材料 |
| 验收层 | QA副主编 | 独立QA Agent；不可用时必须停止当前链路并写`SUBAGENT-BLOCKER` | 冻结/未冻结/阻断 |

## 最小Agent编制

完整HTML/PPT/PDF视觉生产至少需要以下独立审查角色。只开4个“综合审查Agent”不算完整团队。缺少任一适用部门Agent时，不得进入正式文件创建。

| 能力域 | 副主编一级复核 | 执行或专项Agent | 必查内容 |
| :--- | :--- | :--- | :--- |
| 需求与交付 | 需求与交付副主编 | 资料盘点专员 | 项目类型、受众、交付格式、边界 |
| 语言内容 | 语言内容副主编 | 内容结构专员 | 章节逻辑、每页核心观点、术语完整性、术语/缩写表达、中文可读性 |
| 品牌与审美 | 品牌与审美副主编 | 视觉系统专员 | 参考经验转译、流行趋势取舍、反AI味 |
| 版式/UI | 版式/UI副主编 | 跨页一致性专员 | 网格、字号、线条、颜色、表格、组件一致性、光学平衡 |
| 动效设计 | 动效设计副主编 | 动效脚本专员 | 动效目的、节奏、页面角色、分步呈现、首态/后续态、克制/多元边界 |
| 交互与工程 | 交互与工程副主编 | 前端/工具链专员 | HTML行为、导出模式、测试脚本 |
| QA与交付 | QA副主编 | 测试专员 | 全页截图、人工视觉复核、冻结判断 |

局部修复、审稿、QA冻结、Style Frame生成和用户退回回归不得用“小任务”跳过多Agent。最小编制为：受影响能力域副主编Agent、执行Agent或执行记录、QA副主编Agent；若涉及视觉、内容、UI或动效中的任一项，相关能力域副主编Agent也必须参与复审。

## 上传文件后的协同初审

用户上传文件、旧稿、模板或参考素材后，必须先建立并行初审，而不是直接进入页面生产。

| 部门/副主编 | 必须判断 | 提交副总编前条件 |
| :--- | :--- | :--- |
| 语言内容 | 目标、受众、核心信息、缺失内容、术语风险、缩写/模型表达 | 内容边界清晰，术语组件规范明确 |
| 品牌审美 | 参考经验、品牌色、风格风险、反AI味风险 | 审美方向可执行且不照搬参考 |
| 版式/UI | 页面族谱、网格、字阶、表格、线条、对比度 | UI规则可落地且无明显硬伤 |
| 动效设计 | 页面是否需要动、动效目的、分步节奏、导出/演讲风险 | 动效脚本清楚，静止页与动效页边界明确 |
| 交互/交付 | HTML/PPTX/PDF路线、演示模式、导出和测试约束 | 交付路径和降级审批路径明确 |
| QA前置风险 | 截图证据、可读性、素材、冻结条件 | 冻结条件可验证 |

所有适用部门必须在《部门一致意见表》中签署“同意提交副总编”。若存在冲突，相关副主编先协商修订初步方案；副总编和总编只复核已经达成一致的方案，不接收未解决冲突。

## Style Frame协同生成

Style Frame不是品牌审美或版式/UI部门单独出图后的事后审查。生成任一Style Frame截图前，必须先完成《Style Frame部门协同输入表》，并由副总编确认冲突已经解决。缺少协同输入时，只能称为“视觉草图”或“单部门样张”，不得称为Style Frame。

| 部门/副主编 | 生成前必须输入 | 对Style Frame的否决权 |
| :--- | :--- | :--- |
| 语言内容 | 每个样张方向的核心判断、信息载荷、标题/注释/提问句、讲师或使用者动作 | 页面好看但不知道在讲什么、内容密度与讲述目的不匹配 |
| 品牌审美 | 参考经验转译、当前趋势取舍、视觉母线、品牌色与禁用风格 | 像模板、照搬参考、没有项目资料锚点 |
| 版式/UI | 网格、字阶、组件语法、控件状态、表格/卡片/标注规则 | 对齐混乱、层级平均、组件不能跨页复用 |
| 动效设计 | 哪些元素分步出现、为什么动、哪些页必须静止、导出稳定态 | 全页统一动画、动效遮挡阅读、export截到中间态 |
| 交互工程 | HTML/PPT实现方式、响应和投屏约束、截图/导出模式 | 设计无法稳定实现、控件破坏画面、导出不可控 |
| QA前置风险 | 截图有效性、可读性、对比度、重点页人工检查项 | 证据不足、截图空白/半成品、无法验证冻结条件 |

《Style Frame部门协同输入表》必须与截图一起归档。副主编只能签署自己部门的判断，不能代签其他部门；若语言内容、UI、动效或QA提出退回，品牌审美不得单独宣布通过。副总编只接收已经解决冲突的Style Frame，总编只复核已经完成协同输入和截图证据的Style Frame。

截图不得早于协同输入完成。若Style Frame截图生成时间早于《内容到UI/动效转译表》《Style Frame部门协同输入表》或冲突解决记录的完成时间，该截图不得追认为Style Frame，必须在冲突解决后重新生成。QA不得冻结截图时间早于协同输入和动效脚本确认时间的版本。

## 强制运行顺序

1. 主Agent检测是否有可用subagent工具。
2. 在协作记录中写入《Agent编制表》：角色、agent id、职责、输入、输出、状态。
3. 建立《部门一致意见表》：语言内容、品牌审美、版式/UI、动效设计、交互/交付、QA前置风险全部同意后，才提交副总编。
4. 副总编审核部门意见是否真实独立、是否存在冲突、是否需要新增部门；通过后提交总编初步复核。总编通过后进入交付定义、页数充分性、页面族谱、内容到UI/动效转译、模板适配/改造和Style Frame协同输入。
5. 以上生产前判断完成并通过生产前放行后，才下发制作工单；执行层Agent再生产或专项检查，不得签署通过。
6. 对应副主编Agent做一级复核，必须输出退回清单或通过理由。
7. 任一副主编退回时，该副主编Agent必须建立修复任务并下发执行层；执行层修复后回到同一副主编Agent复审。无论多少轮，均保留轮次、修复证据和复审结论。
8. 全部适用副主编Agent一致签署“可交副总编”后，副总编先做一致性审核；副总编签署“可交总编”后，总编才做二级复核。若副主编之间冲突，总编不得直接覆盖为通过，必须退回冲突副主编协商。
9. 总编二级复核若提出修改意见，必须下发给对应副主编Agent，由副主编Agent下发执行层；修复后重新进入副主编复审循环，再一致提交总编。
10. 总编无复核意见后，QA副主编做最终验收。QA拥有冻结否决权；总编不能替QA冻结。
11. QA冻结后提交用户验收，并记录用户验收结论。
12. 用户退回时，总编必须把意见分发给对应副主编Agent，重新进入“副主编下发执行层 → 执行层修复 → 副主编复审 → 全部副主编一致 → 总编复核 → QA回归 → 用户复验”。
13. 记录专员把每轮agent产出、退回、修复和复审写入协作记录。
14. 完成后关闭不再需要的子Agent，避免残留运行。

## 实现后多轮复核

任何视觉产物至少区分四类记录：

| 记录类型 | 输入 | 是否可替代其他环节 |
| :--- | :--- | :--- |
| 生产前输入 | 用户资料、参考库、Page Spec、Style Frame方向 | 不能替代实现后复核 |
| 实现后一级复核 | 真实HTML/PPT/PDF、源码路径、截图、validation、问题页清单 | 不能替代QA冻结 |
| 退回复审 | 修复后的真实文件、修前/修后截图、同类页联动证据 | 不能由主Agent代签 |
| QA冻结 | 全页截图、人工审美QA、工程烟测、冻结/阻断结论 | 不能由自动脚本单独签署 |

最小合格闭环：

1. 每个适用能力域都有生产前输入。
2. 执行层生成后，品牌审美、版式/UI、语言内容、动效、交互工程和QA中的适用副主编至少完成一轮实现后复核。
3. 任一副主编退回时，修复后必须回到同一副主编做复审，不能换人消解责任。
4. QA必须在副主编复审结束后才冻结；QA发现审美硬伤时，退回对应副主编，而不是让执行层直接改CSS。
5. 最终回复必须说明哪些复核已完成；缺失时必须写“未冻结/未交付”。

## 降级审批规则

| 场景 | 处理 |
| :--- | :--- |
| subagent工具可用 | 必须创建对应能力域Agent；不得只在主Agent内角色扮演 |
| 用户已显式提到subagent/副主编/多Agent | 视为已授权创建；必须立即建立能力域Agent，而不是继续主Agent自审 |
| 工具政策要求显式授权但用户尚未提到subagent | 本Skill默认建立真实能力域Agent；不得询问用户许可 |
| subagent工具不可用或权限阻断 | 立即暂停正式生产；写入`SUBAGENT-BLOCKER`；不得继续生成、截图、validation或QA冻结 |
| 必要部门Agent不可创建 | 立即暂停；不得用主Agent临时扮演；写入`SUBAGENT-BLOCKER` |
| 用户已质疑未调用subagent或未按流程图 | 后续同项目必须创建subagent；若工具不可用，只能写`BLOCKED_BY_SUBAGENT`，不得生成下一版、截图包或validation |
| 用户只要求“创建/生成文件”且未指定格式 | 按`03-delivery-routing.md`判断交付路线，不得在多Agent文件内另行定义默认格式 |
| 用户暂停或质疑组织设计 | 立即停止新建/运行Agent，先重设团队结构 |
| Agent产出互相冲突 | 总编不得忽略冲突；必须写入冲突、取舍和复核结论 |
| Agent只给空泛意见 | 退回该Agent，要求按页码、文件、证据、修复方向重写 |
| 后续轮次缺少副主编复审 | 总编不得接收，QA不得冻结；退回补齐复审循环 |
| 只有生产前预审，没有实现后截图复核 | 视为未完成多轮复核；不得交付 |
| 主Agent快速修复用户退回意见 | 暂停生产，进入失败审计；恢复生产需用户明确同意 |

## 禁止行为

1. 只创建几个泛化审查Agent，然后宣称已执行完整团队。
2. 副主编和执行专员混为一个Agent，且没有独立复核。
3. 主Agent同时做生产、一级复核、二级复核、QA冻结，或用“主Agent分角色”替代真实部门Agent。
4. 没有逐页截图或人工视觉证据，就宣称UI/最终产品已核验。
5. 用“流程已记录”替代“流程已运行”。
6. 首轮副主编发现问题后，后续修复只由主Agent或执行层自查。
7. 在副主编Agent未一致同意时，把产物交给总编复核。
8. QA冻结后直接宣称完成，而没有用户验收记录或用户退回闭环。
9. subagent工具不可用时继续生产。
10. 发现需要新部门时未经总编批准、未创建子Agent，就由主Agent临时扮演。
11. 动效、UI、审美、语言内容互相代签，导致复核形同虚设。
12. 在用户已经质疑subagent或副主编参与后，继续以“工具策略未授权”为由不创建Agent。
13. 协作记录里写“咨询/复核/预检通过”，但没有对应Agent id、输入、输出和退回/通过证据。
14. 用“阻断说明”掩盖未执行团队流程，并继续把产物标为候选稿或冻结稿。
15. 把预审意见、Style Frame建议或探索报告当成实现后多轮复核。
16. 用户退回审美或流程后，主Agent绕过失败审计继续创建下一版。
