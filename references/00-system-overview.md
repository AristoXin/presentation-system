# 00-system-overview

读取时机：当任务需要先理解系统定位、适用范围、整体原则，或用户要求审计/解释整个生产系统时读取。

本系统是一套通用presentation / visual storytelling / 多格式视觉交付系统，不是培训课件专用Skill。它覆盖客户汇报、内部管理汇报、销售提案、投融资路演、培训课件、研究报告、产品介绍、工作坊材料、项目复盘、招投标材料、会议讲稿、PDF讲义、HTML互动演示、PPTX可编辑母版、截图包、证据包和speaker notes。

核心原则：

- 先定义沟通任务，再定义交付物。
- 若用户只说“创建文件/生成版本/做一个版本”且未指定格式，演示、汇报、培训、视觉叙事和HTML样张场景默认交付HTML文件；PPTX/PDF/截图包是明确要求或HTML后的衍生产物。
- 先定义交付物，再定义内容结构、语言内容和术语组件。
- 先定义内容结构，再定义UI。
- 先判断页数充分性，再定义页面族谱、Page Spec和设计引擎；页数不是旧版、样本或模板的默认继承项。
- 参考模板决定约束，不决定答案。
- 视觉参考必须沉淀为设计经验，不只作为临时灵感使用。
- 视觉参考和流行趋势只能被转译，不能照搬；每次设计必须结合用户资料、品牌色、受众、场景和真实证据重新生成方案。
- 内容创造部门必须先把核心判断、证据关系和讲述动作转译为UI结构与动效/静止策略，再进入视觉实现。
- 完整视觉生产前先通过Style Frame审美闸门；Style Frame必须由语言内容、品牌审美、版式/UI、动效设计、交互工程和QA共同生成，不得由视觉部门单独出图后事后盖章。
- HTML、分步呈现、转场或动画任务必须先由动效设计部输出动效脚本；动效应服务讲述、理解和控场，不得工程层临时添加。
- 完整HTML演示必须检查光学平衡、首态/后续态矩阵和动效美感；工程可运行不等于视觉可交付。
- 正式生产前必须建立真实团队链路：语言内容、品牌审美、版式/UI、动效设计、交互/交付和QA并行初审，副总编审核一致性，总编再做方案闸门。
- 工具、subagent或必要部门不可用时，不得自动降级；必要工具或Skill缺失时先请求安装/启用或内嵌，用户拒绝则停止；subagent或必要部门Agent不可用时写入`SUBAGENT-BLOCKER`并停止生产/复核链路。
- Skill输出不直接交付，必须验收。

## 使用边界

- 用户未声明格式时，按`03-delivery-routing.md`判断交付路线；本文件不重复定义默认格式。
- 不默认用户资料用于授课或培训，必须先识别项目类型、受众、使用方式和交付格式。
- 不把参考模板等同于最终页数命令，必须先判断模板约束等级。
- 不把用户旧版、样本、模板或历史产物页数当作最终页数，必须由内容、文案、审美、UI、动效、交互/交付和QA共同判断页数充分性。
- 不把参考图等同于最终视觉答案，必须先完成参考经验转译和当前项目绑定。
- 不把旧版本当作新版模板；当用户明确说旧版只是参考或指出模板化时，必须先过旧版否决门。
- 不把外部Skill输出当最终稿，必须有人类标准的审查和QA证据链。
- 不允许主Agent临时扮演新增部门；发现需要新部门时，必须提交新增部门申请，经副总编预审、总编批准并创建独立subagent后才成立。

## 最小产出

任何正式生产前，至少要能回答：

1. 资料用于什么沟通场景？
2. 主要受众是谁？
3. 需要推动什么行动或认知变化？
4. 最终交付格式是什么？
5. 哪些内容、工具、模板或素材存在风险？
6. 是否存在需要沉淀到`15-visual-reference-library.md`的视觉参考、风格理念或调研结论？
7. 若使用过往参考经验，哪些被借鉴、哪些被排除、当前用户资料如何改写它？
8. 是否已有《内容到UI/动效转译表》，证明UI和动效来自内容关系而不是套模板？
9. 是否已有《页数充分性判断表》，证明页数、拆合页、附录或讲者备注由跨部门判断产生？
10. 是否已建立语言内容、品牌审美、版式/UI、动效设计、交互/交付、QA的并行初审和副总编/总编双闸门？
11. Style Frame是否有部门协同输入、冲突解决和截图证据？
12. 若涉及动效，动效脚本是否说明了为什么动、何时动、何时静止、首态/后续态如何触发和如何导出？
13. 若工具、必要Skill或subagent不可用，是否已有阻断记录？联网/浏览器控件是否已提请用户同意安装/启用？subagent不可用时是否已写入`SUBAGENT-BLOCKER`并停止生产/复核链路？
